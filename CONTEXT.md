# Expense Tracker 领域上下文

## 项目目标

本项目是一个用于练习的 Java CLI Expense Tracker。目标是构建一个终端命令行工具，帮助用户在本地记录收入和支出，并查看周汇总和月汇总。

## v1 范围

v1 必须形成以下闭环：

- 用户可以通过命令新增一笔收入或支出。
- 交易数据必须保存到本地文件，程序退出后再次运行仍可读取。
- 用户可以查看交易列表，用于核对已经录入的数据。
- 用户可以删除一笔录错的交易。
- 用户可以查看周汇总和月汇总。
- 系统必须执行基础数据校验。
- CLI 必须提供帮助信息，让用户知道可用命令。

v1 明确不支持编辑已有交易。录入错误时，用户通过删除交易后重新新增来修正。

v1 删除交易采用物理删除：从 CSV 存储文件中移除对应记录，不保留 `deletedAt` 或软删除标记。

## 技术栈

v1 使用 Java 实现，项目构建工具使用 Gradle，测试框架使用 JUnit 5。

Gradle 只承担项目构建、测试运行和依赖管理职责。v1 不因为使用 Gradle 而引入应用框架。

CSV 读写使用 Apache Commons CSV，避免手写不完整的 CSV 解析器。Commons CSV 只能用于 CSV 读写适配层，不能泄漏到领域模型和汇总逻辑中。

标准目录结构：

```text
src/main/java
src/test/java
```

## 版本管理

本项目使用 Semantic Versioning 管理版本，版本号格式为 `MAJOR.MINOR.PATCH`。

版本变动规则：

- 不兼容的 CLI、CSV 数据格式、命令输出契约或公共行为变更提升 `MAJOR`。
- 向后兼容的新增功能提升 `MINOR`。
- 向后兼容的 bug 修复提升 `PATCH`。

对本项目而言，“公共契约”包括用户可调用的 CLI 命令、参数、退出码、用户可依赖的输出语义、CSV 存储格式和既有命令行为。

每次新增特性或修复 bug 时，都必须评估并同步更新项目版本号。纯规划文档修改可以不改变版本号。

## v1 包结构

v1 使用简单分层包结构：

```text
com.example.expensetracker
├── cli
├── domain
├── application
└── infrastructure
```

职责边界：

- `cli`：参数解析、输出文本、退出码。
- `domain`：`Transaction`、`TransactionType`、金额和日期校验、汇总模型。
- `application`：新增、删除、列表、汇总这些用例服务。
- `infrastructure`：CSV 文件读写、ID 生成、系统时间适配。

这个结构用于体现领域逻辑与命令行输入输出分离，不引入复杂架构模式。

## v1 CLI 命令

v1 使用非交互式参数命令。参数缺失或非法时，命令应失败并打印清晰用法说明，不进入交互式问答。

建议命令形态：

```text
expense add --type expense --amount 42.50 --category food --date 2026-06-22 --note "lunch"
expense list
expense delete <id>
expense summary week --date 2026-06-22
expense summary month --month 2026-06
expense help
```

命令规则：

- `add` 用参数创建交易。
- `list` 默认显示最近交易，后续可扩展过滤参数。
- `delete <id>` 物理删除指定交易：从 CSV 存储文件中移除记录，并重写 CSV 文件。
- 删除不存在的 `id` 时命令失败，并提示未找到对应交易。
- 删除成功后输出被删除交易的简短信息，方便用户确认。
- `summary week --date YYYY-MM-DD` 使用某一天定位所在 ISO 周。
- `summary month --month YYYY-MM` 使用年月定位自然月。
- `help` 或无参数时展示可用命令。

## 领域术语

### 交易

一条收入或支出记录。交易至少包含金额、类型、分类、日期和备注。

v1 交易固定包含以下字段：

| 字段 | 含义 | 规则 |
| --- | --- | --- |
| `id` | 交易唯一标识 | 用于删除交易，必须唯一 |
| `type` | 交易类型 | 只能是 `income` 或 `expense` |
| `amount` | 金额 | 必须大于 0，最多 2 位小数 |
| `category` | 分类 | 不能为空 |
| `date` | 交易发生日期 | 用于周汇总和月汇总，必须是合法日期 |
| `note` | 备注 | 可为空 |
| `createdAt` | 记录创建时间 | 用于审计和默认排序 |

### 交易类型

交易的方向。v1 只支持两种类型：`income` 和 `expense`。

### 收入

使用户可用资金增加的交易。CSV 中使用 `income` 表示。

### 支出

使用户可用资金减少的交易。CSV 中使用 `expense` 表示。

### 分类

用户为交易指定的用途或来源标签，例如餐饮、交通、工资。v1 要求分类不能为空。

### 金额

交易的数值。v1 使用 Java `BigDecimal` 表示金额，不使用 `double`。

v1 金额规则：

- 输入金额必须大于 `0`。
- 输入金额最多保留 2 位小数。
- 输出金额统一显示 2 位小数。
- v1 不处理币种，金额是无币种数值。

### 日期

交易发生的日期。v1 要求日期格式合法。

### 备注

用户为交易添加的可选说明。CSV 读写必须正确处理备注中的逗号、引号和换行。

### 周汇总

按 ISO 周统计的收支汇总。v1 中周一是一周开始，周日是一周结束。

`summary week --date YYYY-MM-DD` 使用传入日期定位其所在的周一到周日。周汇总至少包含总收入、总支出、净收入，以及按分类分组的支出小计。

### 月汇总

按自然月统计的收支汇总。v1 至少包含总收入、总支出、净收入，以及按分类分组的支出小计。

### 净收入

总收入减去总支出后的结果。

### CSV 存储文件

v1 的本地数据源。应用启动、列表、删除和汇总都从 CSV 存储文件读取交易数据；新增和删除交易会更新这个文件。

CSV 不只是导入导出格式，也是 v1 的主存储格式。

### 物理删除

从 CSV 存储文件中移除交易记录。v1 删除交易使用物理删除，不使用软删除字段。

## 设计语言

- “领域逻辑”指交易校验、汇总计算、日期范围计算、分类统计等不依赖终端输入输出的代码。
- “CLI 层”指命令解析、参数读取、错误展示和结果打印。
- “本地存储”指保存在用户机器上的 CSV 文件，不依赖数据库或远程服务。
- CSV 解析必须正确处理逗号、引号和换行转义，不能用简单的 `split(",")` 实现。
- CSV 读写封装在 infrastructure 层，领域模型和汇总逻辑不依赖 Apache Commons CSV。
- 金额计算必须使用 `BigDecimal`，避免浮点误差影响汇总。
- 单元测试使用 JUnit 5，重点覆盖解析、校验、汇总和日期范围计算。
- CSV 读写测试必须覆盖备注包含逗号、引号和换行的场景。
- 包结构保持简单分层，避免为 v1 引入复杂架构模式。
- 删除交易使用物理删除；列表和汇总不需要处理软删除过滤。
