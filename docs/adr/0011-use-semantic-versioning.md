# ADR-0011: 使用 Semantic Versioning 管理项目版本

## 状态

已接受

## 背景

Expense Tracker CLI 会逐步新增功能并修复 bug。随着命令、参数、退出码、CSV 存储格式和用户可见行为逐步稳定，项目需要一套明确的版本管理规则，让使用者和维护者理解每次发布的兼容性影响。

Semantic Versioning 提供了清晰的 `MAJOR.MINOR.PATCH` 版本号语义：不兼容变更、向后兼容新增功能和向后兼容 bug 修复分别对应不同版本位。

## 决策

本项目使用 Semantic Versioning 管理版本，版本号格式为：

```text
MAJOR.MINOR.PATCH
```

规则：

- 不兼容的 CLI、CSV 数据格式、命令输出契约或公共行为变更提升 `MAJOR`。
- 向后兼容的新增功能提升 `MINOR`。
- 向后兼容的 bug 修复提升 `PATCH`。
- 新增特性或修复 bug 时，必须评估并同步更新项目版本号。
- 纯规划文档修改可以不改变版本号。
- 只要代码版本号发生变化，完成提交后必须创建对应注解版本 tag，格式为 `vMAJOR.MINOR.PATCH`。
- 推送版本变动时，必须同时推送代码提交和对应版本 tag，并验证 tag 指向该版本最终应发布的提交。

本项目的公共契约包括：

- CLI 命令名称和参数。
- 命令退出码。
- 用户可依赖的输出语义。
- CSV 存储格式。
- 既有命令行为。

## 后果

好处：

- 使用者可以通过版本号判断升级风险。
- 后续新增特性和 bug 修复都有明确版本变动规则。
- 破坏性 CLI 或数据格式变更不会被隐藏在普通修复中。

代价：

- 每次功能或 bug 修复都需要额外判断版本影响。
- 需要在实现项目版本字段后，把版本更新纳入开发流程。
- 对 CLI 输出和 CSV 格式的兼容性判断需要保持严谨。
